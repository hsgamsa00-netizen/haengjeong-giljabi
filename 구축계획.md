# 화성 시민 행정 길잡이 — 구축 계획·설계 원칙·사용자 의도 (v1, 2026-06-29)

> **이 문서의 목적**: 2026 화성시 시민·공무원 AI 공모전 출품작 「화성 시민 행정 길잡이」의 전체 의도·설계·구축 계획을 **자기완결적으로** 정리한다. **다른 AI의 교차검토용**이므로, 사전 지식 없이 이 문서만으로 맥락·근거·결정을 이해할 수 있도록 작성한다. 마지막 18장에 **중점 검토 요청 포인트**를 둔다.

## 1. 공모전 개요 (요건·제약)

- **공모전**: 2026년 화성시 시민·공무원 AI 공모전 (「AI 화성 챌린지」 Part 1). 화성시 공고 제2026–2236호.
- **접수**: 2026.6.29 ~ 7.10. 본행사·발표심사 9.17(예정).
- **주제**: *일상 속 AI 기술을 접목해 시민 편익을 높이는 창의적 솔루션* (교통·복지·안전·행정·환경·문화·교육 / 관광·상권·지역콘텐츠).
- **출품 형식**: **웹 브라우저 접속 URL**(웹앱·대시보드). 별도 설치 없이 Chrome/Edge에서 즉시 시연. **실제 작동(AI·API·공공데이터 연동)이면 기술성·완성도 고배점**. 단순 기획서·실행불가 디자인 불인정.
- **개발 방식**: **바이브코딩**(자연어로 설명 → AI가 코드 생성) 명시 권장.
- **심사 5축(각 20점·총 100)**: ① 창의성 ② 효율성 ③ **적용 범위** ④ **계속성** ⑤ 기술성(완성도).
- **🔴 결정적 제약**: "**단순 건의·진정 등 민원성 제안은 심사 대상에서 제외**" → 본 솔루션은 민원 접수 채널이 아니라 **지식·길안내 서비스**로 설계해야 함.
- **부문**: 화성시민 부문 / **공무원 부문**(화성시 소속 공무원). 출품자는 공무원 부문.
- **유의**: 본인 순수창작물·타 공모전 수상이력 無·저작권/지재권 분쟁 시 응모자 책임·발표자료에 API Key·개인정보·보안정보 금지·수상작 화성시 공익 무상이용.

## 2. 출품자·보유 자산

- **출품자**: 화성시 감사관(공무원). 공공감사 도메인 전문가이자 본 프로그램의 1인 개발자.
- **보유 자산**: 「감사 길잡이」 — 화성 자체감사 공개사례 약 3,400~3,668건을 **빌드타임 AI로 정제(디제스트)**한 웹앱(GitHub Pages 라이브). 카드 31필드(제목·요지·위반사실·분야·위반유형·처분종류·원인·적신호·핵심구·구조·경과 등). **런타임 LLM 0**(클라이언트 검색·필터)·무료·오프라인·PII(실명) ○○○ 마스킹.
- **자산 활용 방침**: 감사 길잡이는 **새 서비스의 'base'가 아니라 '참고 자산'**. 시민-first로 전면 재설계하되, 감사 데이터·UIUX 원칙·빌드타임 패턴을 차용.

## 3. 핵심 문제의식 & 전략 방향 (사용자 의도)

1. **전면 재설계** (리네이밍 금지): 기존 감사 길잡이는 감사 보조·착안점·지적 통계에 치중 → 공모 주제 "시민 일상 직접 도움"과 부합하지 않음. 메뉴·이름만 바꾸는 수준으로는 미달. → **프로그램 자체를 시민-first로 새로 설계**.
2. **시민 일상 직접 도움 (엄격)**: 추상적 투명성이 아니라, 시민이 실제로 손에 쥐는 편익(받을 혜택·절차·권리)이어야 함.
3. **감사관 고유 자산의 정당한 전환**: "행정이 어디서·어떻게 잘못하는지"에 대한 감사 지식을, 시민에겐 **권익·예방의 길안내**로 전환. 단 아래 4 원칙을 지킴.
4. **감사 '선례'의 시민 직접 노출 금지**: 시민에게 감사 지적 선례가 튀어나오면 ① 당혹스럽고 ② **불필요한 감사청구를 유발**(감사관이 피해야 할 역효과). 감사 선례는 본질상 행정을 '하는' 쪽(공직자)에게 유용 → **청중 분리**(10장).
5. **UIUX**: 감사 길잡이의 정제된 UIUX 원칙을 계승하되, **자잘한 텍스트·설명을 최소화하고 큼직한 선택·시각적 스캔 우선**으로 시민 친화화(9장).

## 4. 컨셉 정의

**「화성 시민 행정 길잡이」** — *시민이 화성시 행정과 얽힌 어떤 상황이든 평소 말/선택으로 입력하면, ① 받을 혜택을 찾고 ② 절차·처리 지연을 점검하고 ③ 부당한 처분의 권리·구제 길을 안내받는 비-LLM 웹 서비스.* 분야 = 행정/복지. 비유 = "행정판 개인 비서"(정부24가 '메뉴 보여주고 끝'이라면, 신청·추적·불복까지 개인 사건을 데려다 줌).

**3대 시나리오(엔진)**
- (a) **받을 권리 찾기**: "67세, 혼자 거주" → 놓친 복지·지원금·감면 발굴 + 신청 안내.
- (b) **처리 점검**: "건축허가 65일째 무응답" → 법정 처리기한 점검 + 구제(연장통지·정보공개·고충민원·행정심판) 길안내.
- (c) **부당함 점검**: "주정차 과태료, 응급상황" → 자진납부·약자 감경·이의제기 기한·증빙 안내.

## 5. 연구 근거 요약

세 갈래 웹 리서치(2026-06-29) 결과가 한 점으로 수렴:
- **국내 공백**: 정부24·보조금24·복지로·국민비서·서울톡·법제처 Lawbot·화성in 등은 전부 "조회·알림·FAQ 안내"에서 멈춤. **개인 사건을 맥락 이해해 다음 행동(신청·추적·불복·절차)으로 연결**하는 시민 대면 서비스는 사실상 부재 → 신규성 충분.
- **AI 옴부즈맨 선례**: 부분 기능(권익위 내부분석·구삐 알림·법령검색)만 존재. DoNotPay FTC 제재(2025)로 "AI 변호사·대리" 포지션은 규제 리스크 → **"이해를 돕고 길을 안내"** 포지션이 안전(이것이 동시에 '민원성 제외' 함정도 회피).
- **수상 공식**: 반복 고충 → 공공데이터 + 독점 도메인지식 → **라이브 작동 데모** → 즉시 체감. 화성 감사·행정 데이터는 **타 출품작에 없는 독점 자산**.

## 6. 핵심 설계 판정 (3대 질문 답)

### Q1. 감사사례 비중 — 약 15~25%, 특정 장면 집중 (법령·조례가 75%+ 1차 골격)
- **법령·조례 = 1차 기준**(무엇이 정답인가·시의성). **감사사례 = 2차 레이어**(실제 어디서 틀어지나·선례).
- 기능별 비중: 혜택발굴 0~5% · 처리지연 30~40% · 부당처분·불복 55~65% · 인허가 예방 55~65%.
- 단, **시민 화면에는 감사 선례를 직접 노출하지 않음**(4·10장 정책). 시민에겐 backend 품질 + "흔히 놓치는 점" 도움팁으로만. 선례 직접 노출은 공직자 모드.

### Q2. 시민편익 범위 — "전 시민 × 전 행정접점" 진입 라우터
- 넓게 가능: 복지·지원금·감면 / 세금·과태료(불복) / 민원처리·지연 / 보육·교육 / 환경.
- 일부 가능(시 조례·내부 API 연계 필요): 생활불편(도로·주정차·쓰레기) / 인허가·건축 / 교통·안전.
- 초기 제외: 개인 소송 대리(법률자문 영역).
- 우선 톱3: ① 복지·감면 ② 세금·과태료 불복 ③ 민원처리·지연 추적. 확장 로드맵 = 계속성 논거.

### Q3. 비-LLM(검색·규칙) 구현 — 3대 시나리오 전부 가능, LLM은 선택
- (a) 혜택 = 폼 + **자격 규칙엔진**(보조금24 방식) / (b) 처리지연 = **처리기한 DB + 날짜계산** / (c) 과태료 = **절차 DB 키워드·패싯 검색**. 전부 **런타임 LLM 0**.
- 자유발화 입력만 선택적 LLM — gov.uk식 **가이드 폼(순차 질문)**으로 LLM 없이도 대체 가능.
- 근거: 미 정부 실험상 수치 임계값 규칙은 LLM이 완전성 60% 미만·문서 없는 규칙 0% 정확 → **결정론 코드가 더 정확·안전**.

## 7. 시스템 아키텍처 (4레이어)

| 레이어 | 내용 | 비고 |
|---|---|---|
| **L1 빌드타임 디제스트** | 법령·조례·감사사례·절차를 빌드 시 AI로 구조화 → **사람(감사관) 검증 게이트** → JSON 인코딩 | 정적 자산 |
| **L2 시의성 동기화** | 법제처 국가법령정보 OPEN API(`target=ordin`·`org`=화성·`시행일` 델타)로 조례·법령 현행성 일/주 배치 갱신 | **유일한 신규 레이어** |
| **L3 런타임 검색·규칙** | 클라이언트 전문검색(Fuse.js류)+자격 규칙엔진+날짜계산. **런타임 LLM 0·무료·오프라인** | 핵심 동작 |
| **L4 선택적 경량 LLM** | 자유발화 → 구조화 입력 변환만(가이드 폼 폴백 병행) | 선택 |

- **데이터 소스/SSoT**: 화성 감사·행정 데이터 한 벌(기존 감사 길잡이 `cards_data.js` 가공). 시민 모드·공직자 모드가 공유.
- **현행성 표기**: 카드에 `시행일·공포일·출처(법제처)` 표기. 시행 전 개정안은 "예정" 배지.

## 8. ★ UIUX 원칙 (감사 길잡이 계승 + 큼직·간결 강조)

> **사용자 지시 핵심**: 가급적 **자잘한 텍스트·설명을 줄이고, 큼직큼직한 선택과 이용자 눈에 잘 들어오는 방식**으로 개발할 것.

### 8.1 감사 길잡이에서 계승할 원칙
- **정제 미니멀**: 그라데이션·과한 장식 배제, 플랫·정돈된 표면, KRDS(한국 정부 디자인 시스템) 정합.
- **2영역 단일 동선**: 선택→결과의 명확한 단일 흐름. 곁가지 최소.
- **점진적 공개(progressive disclosure)**: 처음부터 다 보여주지 않고 단계별로. 상세는 클릭/오버레이.
- **색 = 경중 1축**: 의미색은 상태(정상/주의/위험·green/amber/red)에만. 장식색 금지.
- **타이포 토큰**: 정해진 글자 크기 단계만 사용(자의적 폰트 크기 난립 금지). 굵기 2단(400/700 또는 400/500).
- **아이콘**: 이모지 금지 → **라인 아이콘**(Lucide/Tabler 계열) 통일.
- **버튼·포커스**: 주행동은 채움 버튼, 보조는 아웃라인. `:focus-visible` 포커스 링(접근성).
- **모바일 우선**: 반응형, 한 손 스캔, 큰 탭 타깃.

### 8.2 시민 앱 특화 강화(이번 지시 반영)
- **큼직한 선택 중심**: 작은 링크/메뉴 대신 **큰 카드·큰 버튼**으로 결정. 탭 타깃 충분히 크게.
- **텍스트 최소화**: 설명 문단 대신 **아이콘 + 짧은 라벨 + 큰 숫자**. 부연은 접기/툴팁/2차 화면으로.
- **한 화면 한 결정**(gov.uk one-thing-per-page): 한 번에 하나만 묻고 고르게.
- **시각 위계 명확**: 핵심 숫자·결과를 크게, 근거·주석은 작게 하단. 스캔 한 번에 "내가 무엇을 얻는지" 파악.
- **분야별 분포형 선택**(11장): 감사 길잡이의 '분야별 분포' 카드 그리드 패턴을 시민 분야로 차용 → 한눈에 고르기.
- **결과는 카드·단계(step)·체크리스트**로 구조화. 장문 회피.

## 10. 청중 분리 & 이중 모드

- **한 앱 + 최초 실행 청중 선택**(localStorage 기억·헤더에서 전환). 잘못 골라도 즉시 변경.
- **두 모드**

| | 시민·민원인 모드 | 공직자 모드 |
|---|---|---|
| 대상 | 시민·민원인 | **공직자**(공무원·공공기관·산하기관 임직원) |
| 기능 | 분야 선택 → 받을권리·처리점검·부당점검 | 분야·위반유형·처분 검색 → 착안점·원인·유사선례(=기존 감사 길잡이 기능 통합) |
| 감사사례 | **비노출**(backend 품질 + "흔히 놓치는 점" 도움팁) | **그대로 노출**(예방 점검·선례 학습) |
| 유도 행동 | 신청·이의제기·고충민원(개인 구제) | 사전 예방·적정 처리 |

- **용어 결정**: 비-시민 청중 포괄어 = **"공직자"**(부제 *공무원·공공기관·산하기관 임직원*). 공직자윤리법상 공직유관단체 임직원까지 포함하는 정확한 상위어. ("공무원"만으론 산하 출자·출연기관 임직원을 못 아우름.)
- **데이터 SSoT 공유**, 감사 선례는 **공직자 모드에서만** → 시민 당혹·불필요한 감사청구 유발 0.
